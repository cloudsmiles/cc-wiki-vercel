# 慢查询日志&开启优化

# 如何开启慢查询日志

### 1、查看慢查询日志是否开启

执行命令：show variables like 'slow%'。

得到以下结果：

![Untitled](RPG从零开始/Database/MySQL/Optimization/慢查询日志&开启优化/Untitled.png)

### 2、开启慢查询日志

执行命令：set global slow_query_log = on。

然后再查询一下，发现slow_query_log处于开启状态：

![Untitled](RPG从零开始/Database/MySQL/Optimization/慢查询日志&开启优化/Untitled%201.png)

### 3、慢查询的临界值设定

首先查看一下默认的临界值。

执行命令：show variables like '%long%'。

![Untitled](RPG从零开始/Database/MySQL/Optimization/慢查询日志&开启优化/Untitled%202.png)

其中有一个long_query_time属性，它的值为10.000000。它表示的意思是，只记录查询时间在10s以上的语句。

显然10s我们是不可接受的，所以我们需要自己设定一下这个值。因为我自己的测试表中只有10w条数据，查询很快，所以这里我们设置的小一点。如果有条件的话，最好设置一个百万级的表进行测试。

我们把慢查询的临界值设置为0.02：set long_query_time=0.001。

![Untitled](RPG从零开始/Database/MySQL/Optimization/慢查询日志&开启优化/Untitled%203.png)

现在来模拟查询时间小于等于0.001查询，看是否记录在了慢查询日志中。

![Untitled](RPG从零开始/Database/MySQL/Optimization/慢查询日志&开启优化/Untitled%204.png)

然后看一下日志文件中的数据：

![Untitled](RPG从零开始/Database/MySQL/Optimization/慢查询日志&开启优化/Untitled%205.png)

需要注意的是，我们上面的操作是在交互界面进行的，如果数据库进行重启，这些设置都会失效。如果要永久生效，需要修改配置文件：

```c
vi /etc/my.cnf
[mysqld]
slow_query_log = 1
long_query_time = 0.1
slow_query_log_file =/usr/local/mysql/mysql_slow.log
```

在配置文件中加上这三行就可以了。主要要重启mysql才能生效!

# 开启性能详情

如果通过上面的语句解析没有定位到问题，我该加的索引也加了，但是还是比较慢，那就可以通过性能详情来进一步的探究一下原因。

性能详情可以追踪查询语句的整个生命周期的状态，有了这些状态值，就可以从更深层次找出具体是哪个环节慢了，从而能针对性的进行改善。

### 1、查看性能详情是否开启

执行命令：show variables like '%profiling%'。

![https://s9.51cto.com/oss/202205/17/916edec276563717a8577633c232e71854ed77.png](https://s9.51cto.com/oss/202205/17/916edec276563717a8577633c232e71854ed77.png)

可以看到profiling属性值为OFF，表示关闭，那么我们先开启它。

执行命令：set profiling = on 。

![https://s4.51cto.com/oss/202205/17/87ab1f76129793081c5681dff329186d714148.png](https://s4.51cto.com/oss/202205/17/87ab1f76129793081c5681dff329186d714148.png)

这样就开启了。开启之后，我们就可以执行查询语句，mysql会自动的保存性能记录。

### 2、查看性能记录

我们执行一条sql语句：

![https://s9.51cto.com/oss/202205/17/d70e4fa271c107d222a378f5c32c170bc2e731.png](https://s9.51cto.com/oss/202205/17/d70e4fa271c107d222a378f5c32c170bc2e731.png)

然后查看性能记录：

执行命令：show profiles。

![https://s9.51cto.com/oss/202205/17/84ac4d27744608ddaf20994fb03a16f3e16712.png](https://s9.51cto.com/oss/202205/17/84ac4d27744608ddaf20994fb03a16f3e16712.png)

可以看到开启后的所有查询语句的记录。我们想查看一下第二条执行语句的整个执行周期的状态详情：

执行命令：show profile for query 2。

![https://s3.51cto.com/oss/202205/17/a1e59fb41f0ea4e12a5824091b4ac485884140.png](https://s3.51cto.com/oss/202205/17/a1e59fb41f0ea4e12a5824091b4ac485884140.png)

可以看到整个执行过程每个状态的耗时情况。然后定位到具体是哪个状态最耗时，然后针对性的排查原因。
