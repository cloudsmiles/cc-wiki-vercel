# 合并报表方案

# 需求背景

## 1.1 目标

1，在boss维护集团公司信息，以便于对股权架构进行管理，进行报表合并

2，对财务科目、分析科目进行维护，并能规划其对应关系，以便于，可以进行科目类型分析

3，在boss系统完成每月合并报表工作，根据目前规划，先行在boss上完成合并资产负债表及合并利润表；

4，建立财务会计科目与分析报表科目的映射关系，初步建立财务分析体系；

[合并报表｜需求2023.03.31](https://zegocloud.feishu.cn/wiki/wikcna6ToDMuRMw5MxHNhknD4ah)

## 1.2 基本概念

### 公司

**集团公司**

最上级的公司

**子公司**

集团公司的子公司，如果是最末级公司，无法进行合并报表操作

### 科目

**财务（资产）科目**

会计术语，资产科目是提供总括性的核算指标,如“固定资产”、“原材料”、“应收账款”、“应付...经济用途等。其中利润表的科目与负债表的科目是不一致的

**分析科目**

公司内部使用，用做于内部统计科目余额信息，与财务科目有映射关系

### 报表

**资产负债表**

会计术语，财务状况表，包含资产，负债，所有者权益，其他综合收益

**利润表**

会计术语，反映企业实现的利润或发生的亏损情况

**科目余额表**

会计术语，是每个会计科目的结算余额，它是根据汇总的凭证而形成的每个科目的余额。**目前是每间公司在平台上上传**

**调整分录**

会计术语，目的是把相应的收入和费用金额分配到每个[会计期间](https://baike.baidu.com/item/%E4%BC%9A%E8%AE%A1%E6%9C%9F%E9%97%B4/8601476)。这些分录调整了各种分类账户的余额，因此叫[调整分录](https://baike.baidu.com/item/%E8%B0%83%E6%95%B4%E5%88%86%E5%BD%95/122908)。

**借方，贷方**

会计术语，记账的一个符号，与中文的增减收入的理解有差异

**单项底稿**

**每间公司在平台上进行上传的报表**，形成临时的报表

**单项报表**

在单项底稿的基础上，新增调整记录，形成最终确认的单项报表，单项报表=单项底稿+调整分录

**合并表底稿**

**通过本公司和子公司的报表（可能是单项报表也可能是合并报表）进行合并后的结果**，形成临时的报表

**合并报表**

在合并表底稿的基础上，新增调整记录，形成最终确认的合并报表，合并报表=合并报表底稿+调整分录

### 财务分析

**月度报表**

每个月进行合并报表，产生的最后报表成为月度报表。

月度报表 = 每间公司的单项报表(单项底稿+调整分录)+每间公司的科目余额表+每间公司的合并报表(合并底稿+调整分录)

**分析科目统计**

根据上传的科目余额表，以及分析科目和财务科目的映射关系，计算出分析科目的结算金额。仅根据关联的最末级科目进行合并计算

## 1.3 重点解决问题

公司管理和科目管理的层级结构如何设计？需要支持的功能如下：

- 支持列表展示
- 支持快速查询本节点的所有子节点
- 支持快速删除本节点与递归子节点
- 支持查找父节点

月度报表如何关联公司信息？需要支持的功能如下：

- 当前公司架构变动，不影响历史月度表数据
- 当前月度表生成中，公司架构变动，可以影响月度表数据

月度报表设计数据库如何设计？需要关注的功能如下：

- 以每月为维度做区分
- 公司和科目的信息变动不影响历史已经生成的月度报表（已假设科目信息不会变动）
- 包含创建时的公司信息
- 包含每个公司的单项报表，合并报表，科目余额表
- 对于每个公司的单项报表/合并报表/科目余额表，关联调整汇总表和各自的科目列表

报表的数据库如何设计？需要关注的功能如下：

- 汇率换算需要注意的问题
- 合并报表，如何合并的问题
- 是否存储汇总的冗余信息
- 是否存储科目和公司的冗余信息
- 是否区分每种类型的表来存储

## 2.5 通用报表

设计思路

- 参照“数据架构图”，通过报表ID关联相关科目
- 通过sheet_type和sheet_finance_type的类型判断，形成各种报表

| sheet_type | 0 | 0 | 1 | 1 | 0 |
| --- | --- | --- | --- | --- | --- |
| sheet_finance_type | 0 | 1 | 1 | 1 | 2 |
| 报表类型 | 单项利润表 | 单项资产负债表 | 合并利润表 | 合并资产负债表 | 单项科目余额 |
- **不存储科目和公司的冗余信息，只存储关联的ID**，因为考虑到公司是月度存储，可以绑定关联关系；科目是建立后就不会更改的属性，所以不需要考虑它的变动带来的影响
- **不存储汇总和调整的冗余信息，合并报表的汇总信息通过实时计算得出，不需要存储合并报表的报表信息**

主要流程

- 单项报表上传（包含科目余额）
    - 上传条件检查，科目过滤（不符合条件的过滤并错误提示）
    - 如果没有旧的报表，创建新的报表
    - 如果有旧的报表，删除关联的科目
    - 创建新的科目列表并关联
- 单项报表查看
    - 查找报表关联的调整信息
    - 通过调整计算，得到单项报表的数据
- 合并报表查看
    - 获取每个公司的单项报表信息
    - 获取合并报表
        1. 递归查找本公司和子公司单项报表
        2. 进行报表合并
        3. 查找关联的合并调整，进行调整
    - 不同公司的币种需要换算当前合并报表的币种
- 汇率换算场景
    - 币种变更解决 --- 金额信息返回两种币种
    - 资产负债科目的汇率计算有特殊规则 --- 后端进行计算返回
    - 合并报表，不同公司币种的金额需要换算才能合并
    - 分析科目统计，不同公司币种的金额需要换算

具体数据库设计如下：

```sql
CREATE TABLE `consolid_sheet` (
```

## 2.6 利润科目

设计思路

- 利润科目，需要关注的金额是本年累计金额，经过借贷方计算，得出本年累计调整后的金额

主要流程

- 报表上传，本期金额计算
    - 参照计算公式，查找上月月度报表，同公司的科目信息进行对照

涉及的金额计算公式：

- 本期金额 = 这个月上传的累计金额 - 上个月上传的累计金额
- 本年累计金额，调整后 = 上传的本年累计金额 + 调整的金额
- 合并表的合计数 = sum(每间公司的本年累计金额)
- 合并表的合并数 = sum(每间公司的本年累计金额，调整后)
- 合并表的抵消借方 = sum(每间公司的借方金额)
- 合并表的抵消贷方 = sum(每间公司的贷方金额)
- 合并表的本年累计金额 = 合并表的合并数 + 调整金额

具体数据库设计如下：

```sql
CREATE TABLE `consolid_income_subject` (
  `id` int(10) unsigned NOT NULL AUTO_INCREMENT,
  `subject_id` int(11) unsigned NOT NULL COMMENT '科目表id',
  `sheet_id` int(11) unsigned NOT NULL COMMENT '报表id',

  `last_year_period_amount` decimal(10,2) NOT NULL COMMENT '上年同期金额',
  `current_period_amount` decimal(10,2) NOT NULL COMMENT '本期金额',
  `cumulate_amount` decimal(10,2) NOT NULL COMMENT '本年累计金额',
  `created_at` bigint(20) unsigned NOT NULL DEFAULT '0' COMMENT '创建时间',
  `updated_at` bigint(20) unsigned NOT NULL DEFAULT '0' COMMENT '更新时间',
  `deleted_at` bigint(20) unsigned NOT NULL DEFAULT '0' COMMENT '删除时间',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='利润科目表';

```

## 2.7 资产负债科目

设计思路

- 资产负债科目，需要关注的金额是期末余额，经过借贷方计算和变动数，得出期末余额调整后的金额

主要流程

- 变动数更新
    - 更新对应科目信息
- 资产负债的汇率计算
    - 资产的汇率，期初=年初余额*年初汇率 期末=期末余额*当月期末汇率
    - 负债的汇率，期初=年初余额*年初汇率 期末=期末余额*当月期末汇率

涉及的金额计算公式：

- 期末余额，调整后 = 上传的期末余额 + 调整的金额
- 所有者权益
    - 期初金额 = 科目表的期初
    - 期末余额 = 年初余额+变动数
- 其他综合者权益
    - 期初金额 = 科目表的期初
    - 期末余额 = 资本-负债-所有者权益
- 合并表的合计数 = sum(每间公司的期末余额)
- 合并表的合并数 = sum(每间公司的期末余额，调整后)
- 合并表的抵消借方 = sum(每间公司的借方金额)
- 合并表的抵消贷方 = sum(每间公司的贷方金额)
- 合并表的期末余额 = 合并表的合并数 + 调整金额

具体数据库设计如下：

```sql
CREATE TABLE `consolid_asset_subject` (
  `id` int(10) unsigned NOT NULL AUTO_INCREMENT,
  `subject_id` int(11) unsigned NOT NULL COMMENT '科目表id',
  `sheet_id` int(11) unsigned NOT NULL COMMENT '报表id',

  `year_open_balance` decimal(10,2) NOT NULL COMMENT '年初余额',
  `close_balance` decimal(10,2) NOT NULL COMMENT '期末余额',
  `variate_amount` decimal(10,2) NOT NULL COMMENT '当期变动数',
  `created_at` bigint(20) unsigned NOT NULL DEFAULT '0' COMMENT '创建时间',
  `updated_at` bigint(20) unsigned NOT NULL DEFAULT '0' COMMENT '更新时间',
  `deleted_at` bigint(20) unsigned NOT NULL DEFAULT '0' COMMENT '删除时间',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='资产负债科目表';

```

## 2.9 调整分录

主要流程

- 单项表/合并表调整
    - 新增调整汇总记录（如果存在，则不覆盖）
    - 与旧有的调整科目进行比较，分成三类，
        - 新增列表，新增调整科目
        - 删除列表，删除旧有的调整科目
        - 更新列表，更新旧有的调整科目
    - 如果有附件，进行科目的关联和失联
- 调整文件上传
    - 利用gin框架写上传文件接口，此接口只保存附件，不关联科目

具体数据库设计如下：


## 2.10 分析科目统计

# 评审修改建议

2023年6月2日内部设计方案评审，修改点如下：

- is_delete字段改成delete_time
- 时间存储由datetime格式改成timestamp格式
- 金额存储由int格式改成decimal格式
- 月度表关联公司方案确定
- 取消存储两种币种的冗余字段

- 文档“自增id”备注需要标明清楚，涉及接口AddSubCompany，AddSubSubject
- 科目管理，是否考虑分页 - 暂时不考虑
- 调整记录，需要增加分页
- 详情页需要增加两个状态，是否可以发起调整，是否可以同步架构
- 资产负债表合并详情页，最后三个字段跟利润表合并详情页意义一致
- 更改报表状态的操作接口统一成一个
- 调整的科目信息，需要增加original_file_name字段
- 保存调整接口，需要存储file_id
