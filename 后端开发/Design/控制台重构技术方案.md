### 历史问题

以前model代码的编写没有规范，而且一直使用了gormV1版本，没有使用上新的特性，所以有以下问题

1. 方法封装存在两种风格，一种是struct方法，另外一种是函数，风格没有一致
2. 基本的CRUD缺少规范指引，不同model的代码，理解不一样，加重了维护成本
3. 创建事务时不能复用已有的方法，由于方法引用的是全局变量db，与事务的db对象不一致，所以无法复用
4. 读写分离代码过于冗余，需要指定主从库，gormV2提供读写分离插件，更加简洁和方便
5. 关联查询通过Joins联表，代码复杂不通用，而且污染了model字段，让model增加了很多冗余字段
6. 对于高级查询，由于入参和出参不同，产生了很多查询方法，无法很好的内聚成通用的方法
7. 存在很多判断枚举值的公有函数，这种判断函数变多了之后，使用方难以辨别它们的用法

### struct方法封装

针对第一点，约束所有的CRUD只能封装成struct方法，新的文件存放在entities目录下，通常一个table一个文件，文件命名使用驼峰，如下图：

![Untitled](后端开发/Design/控制台重构技术方案/Untitled%203.png)

如果单个文件太大，struct字段太多需要切分，可以新建文件，文件命名格式遵循

`{tableName}_{moduleName}.go`，比如serviceInfo可以划分出serviceInfo_bill，代表是serviceInfo的子集

优点：理解简单，代码简单。直接在struct实现方法，与旧的方式类似，而且统一规范，降低理解成本

缺点：文件变多了，会导致搜索不方便

### 支持事务

> https://gorm.io/zh_CN/docs/transactions.html
> 

针对第三点，需要兼容事务，通常使用嵌套事务方式，如果想复用已有方法，可以在已有方法增加参数，通过context传递或者直接传gorm.DB

### **读写分离插件**

> https://gorm.io/zh_CN/docs/dbresolver.html
> 

针对第四点，使用读写分离插件

```
// ❌ 旧的方式
db := common.GlobalBackendDB.GetMaster()
db := zdb.GetNonTransactionDatabases(dbs, true, common.GlobalBackendDB)

// ✔️ 使用插件方式
db, _ := gorm.Open(mysql.Open(master), &gorm.Config{})
db.Use(dbresolver.Register(dbresolver.Config{
    Sources:  []gorm.Dialector{mysql.Open(master)},
    Replicas: []gorm.Dialector{mysql.Open(slave1), mysql.Open(slave2)},
// sources/replicas load balancing policy
    Policy: dbresolver.RandomPolicy{},
// print sources/replicas mode in logger
    TraceResolverMode: true,
}))

// 使用 Write 模式：从 sources db `db1` 读取 user
db.Clauses(dbresolver.Write).First(&user)

// 指定 Resolver：从 `secondary` 的 replicas db `db8` 读取 user
db.Clauses(dbresolver.Use("secondary")).First(&user)

// 指定 Resolver 和 Write 模式：从 `secondary` 的 sources db `db6` 或 `db7` 读取 user
db.Clauses(dbresolver.Use("secondary"), dbresolver.Write).First(&user)

```

对比旧的方式，新的方式对于主从库切换是无侵入的，实现原理是利用gorm的callback，进行拦截判断，自动切换，也支持手动切换

优点：代码简单，少写重复的冗余代码

缺点：与旧代码不兼容，只支持V2版本，如果旧代码有查询使用主库，替换的代码需要手动切换。所以替换存在风险

### **关联方式查询**

> https://gorm.io/zh_CN/docs/belongs_to.html
> 

针对第五点，使用gorm提供的关联查询方式

1. gorm支持的关联方式有：
    1. Has one
    2. Has many
    3. Many to many（需要有关系表）
2. **对于Many To Many的关系，关系表的model不应该再设置关联模式**
3. 想要联表查询时，可以使用InnerJoins/Joins方法，只要model存在关联，gorm就能自动补充联表语句
4. 非联表查询，使用Preload方法

```
// ❌ 旧的方式
db.Table("appid_cdn_config as acc").
        Select("acc.*").
        Joins("INNER JOIN service_appid ON acc.app_id = service_appid.app_id").
        Joins("INNER JOIN service_info ON service_appid.s_id = service_info.s_id").
        Where("service_info.domain in (?)", component.GetDomains()).Scan(&acc)

// ✔️ has one
type UserInfo struct {
    UserId                 uint64
    Username               string
    Extra                  *UserInfoExtra `gorm:"foreignKey:UserId;references:UserId"`
}

type UserInfoExtra struct {
    UserId                 uint64
    GuideStatus            int64
}

// ✔️ has many
type UserInfo struct {
    UserId                 uint64
    Username               string
    HistoryList            []*LoginHistory `gorm:"foreignKey:UserId;references:UserId"`
}

type LoginHistory struct {
    Id             int64
    UserId         uint64
}

// ✔️ many to many
type UserService struct {
    ID        int32
    UserId    int32
    SId       int64
}

type UserInfo struct {
    UserId                 int32
    Username               string
    ServiceInfos            []*ServiceInfo `gorm:"many2many:user_service;foreignKey:UserId;joinForeignKey:UserId;References:SId;JoinReferences:SId"`
}

type ServiceInfo struct {
    SId                    int64
    ServiceType            int32
    Users                  []*UserInfo `gorm:"many2many:user_service;foreignKey:SId;joinForeignKey:SId;References:UserId;JoinReferences:UserId"`
}

// 加载关联数据
// Preload方式
db.Preload("Admin").Find(&ServiceInfo{})
// select * from service_info
// selcet * from user where user_id = ?

// 联表方式
db.InnerJoins("Admin").Find(&ServiceInfo{})
// select service_info.*, user.* from service_info inner join user on service_info.admin=user.user_id

```

相比于旧的方式，通过model就能直观看到关联关系，在涉及到关系表的联表查询时，也节省了很多代码量，通过Preload/Joins/InnerJoins方法就能直接加载数据，而使用旧的方式就需要写原生SQL语句

优点：节省了代码量，关联查询变得简单。而且不会污染model字段，按需加载

缺点：需要model维护关联关系，在涉及多张表联表时，可能反而没原生SQL直观

### **scopes方法**

> https://gorm.io/zh_CN/docs/scopes.html
> 

gorm提供了option的方式封装，对于通用的逻辑可以使用这种方式扩展

```
// ❌ 旧的方式
// 此方法无法扩展，因为判断条件已经写死
func GetStopAppListUtil() ([]*TableServiceAppId, error) {
    db := common.GlobalBackendDB.GetSlave()
    db = db.Table("service_appid")

    sql := "select app_id,status from service_appid where status = 2 and s_id in( select s_id from service_info where status = 2 and domain =1 )"

    var tableListRec []*TableServiceAppId
    err := db.Raw(sql).Scan(&tableListRec).Error
    return tableListRec, err
}

// ✔️ scopes方法
func Smb(db *gorm.DB) *gorm.DB {
  return db.Where("client_version = ?", 3).Where("status = ?", 1)
}

func FindUser(users []int32) func (db *gorm.DB) *gorm.DB {
  return func (db *gorm.DB) *gorm.DB {
      return db.Where("user_id in (?)", users)
  }
}

// 查找正常状态的SMB客户,user_id为12
db.Scopes(Smb, FindUser(12)).Find(&customs)
// 匿名函数进行查找
db.Scopes(func(db*gorm.DB) *gorm.DB {
    return db.Where("user_id=?", 12).Where("client_version=?", 3)
}).Find(&customs)

```

对于一些通用的逻辑操作，比如固定的状态、分类、标记，条件限制，分页，排序，都可以通过option方式封装，也可以作为扩展方法的手段

优点：可以对原有的方法增加功能，而且无侵入。通用的逻辑也可以进行封装，减少重复代码

缺点：需要独立封装方法，对于外部调用者会增加理解，对于内部需要额外维护这些方法

### 钩子方法

> https://gorm.io/zh_CN/docs/hooks.html
> 

gorm提供了一套钩子方法，直接在model实现方法即可

创建对象

- BeforeSave
- BeforeCreate
- AfterCreate
- AfterSave

更新对象

- BeforeSave
- BeforeUpdate
- AfterUpdate
- AfterSave

删除对象

- BeforeDelete
- AfterDelete

```
// ✔️ hook方法
type ServiceInfo struct {
    SId                 int32
    ServiceType         int32
}

func (u *ServiceInfo) BeforeCreate(tx *gorm.DB) {
    sid := generateSid()
    u.sid = sid
}

```

如上述代码，在创建前先执行生成sid方法，形成一套通用规则，可以减少重复代码

优点：减少重复代码，形成约束规则，有利于代码维护

缺点：规则不一定通用

### 枚举值判断

针对第七点，统一规范枚举值判断为model的方法

```
// ❌ 旧的方式
// IsManualCustom 是否人工客户
func IsManualCustom(clientVersion int) bool {
    return clientVersion == ClientVersionManual
}

// IsAutoCustom 是否自助客户
func IsAutoCustom(clientVersion int) bool {
    return clientVersion == ClientVersionAuto
}

// ✔️
func (t *ServiceInfo) IsManualCustom() bool {
    return t.ClientVersion == ClientVersionManual
}

func (t *ServiceInfo) IsAutoCustom() bool {
    return t.ClientVersion == ClientVersionAuto
}

```

优点：减少公有方法的数量，调用方不需要辨别具体的字段，只需理解函数的功能含义即可

缺点：需要先查询具体的数据

### 最佳实践

针对第二点和第六点，需要有个通用的规范，指引我们编写CRUD代码，以下以UserInfo为例子说明

```
type UserInfo struct {
    UserID                 int32  `gorm:"column:user_id;type:int(11);primaryKey;autoIncrement:true;comment:用户ID" json:"user_id"`
    Bid                    string `gorm:"column:bid;type:char(128);comment:业务关联的用户ID" json:"bid"`
    Type                   string `gorm:"column:type;type:char(32);comment:业务类型" json:"type"`
    Username               string `gorm:"column:username;type:varchar(255);not null;comment:用户名" json:"username"`
    Email                  string `gorm:"column:email;type:varchar(255);not null;comment:邮箱" json:"email"`
    EmailStatus            uint32 `gorm:"column:email_status;type:tinyint(3) unsigned;not null;comment:邮箱状态" json:"email_status"`
    Mobile                 string `gorm:"column:mobile;type:varchar(45);not null;comment:手机" json:"mobile"`
}

// 自定义方法
func (u *UserInfo) GetUserList() ([]*UserInfo, error) {
    var result []*UserInfo
    err := DB().Find(&result)
    return result, err
}

//------------------ CRUD使用 ------------------

// 创建
us := &UserInfo{
    Bid: "bid",
    Type: "type",
    Username: "usernam123"
}
users := []*UserInfo{
    {Bid: "bid"},
    {Bid: "bid2"},
}

// 单个创建
DB().Create(us)

// 批量创建
DB().Create(users)

// 更新
us := &UserInfo{
    UserID: 1,
    Bid: "bid",
    Type: "type",
    Username: "usernam123"
}
// 根据主键 非零值更新
DB().Updates(us)

// 根据主键 零值更新
DB().Select("mobile", "email").Updates(us)

// 批量更新 非零值更新
DB().Where("mobile=?", 123).Updates(us)

// 批量更新 零值更新
DB().Select("mobile", "email").Where("mobile=?", 123).Updates(us)

// 删除
us := &UserInfo{
    UserID: 1,
    Bid: "bid",
    Type: "type",
    Username: "usernam123"
}
users := []*UserInfo{
    {UserID: 1},
    {UserID: 2}
}

// 单个删除
DB().Delete(us)

// 多个删除
DB().Delete(users)

// 查询
us := &UserInfo{
    UserID: 1,
}

// 根据主键查询
DB().First(us)

// 批量查询
var users []*UserInfo

DB().Where("mobile=?", 123123).Find(&users)

```

对于复杂的场景，通用的功能，应该封装在model的方法内，基本的CRUD和批量CRUD，应该直接调用db对象进行操作，而且倾向于使用model作为入参，节省代码量
