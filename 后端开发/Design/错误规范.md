# 错误规范

> 在日志规范中有对错误规范制定了一些标准，但还不能完全覆盖控制台的错误规范问题，比如对错误码的处理没有统一标准；fmt.Errorf, errors.Errorf, errors.New等方式抛出error，可能会失去原本error的其他信息；调用外部系统返回的错误，需要适配控制台，返回特定的错误信息，当前没有统一的解决方案
> 

## 错误建立方式

1. 基于字符串

基于字符串，抛出错误信息，以下有两种使用方式

| 静态字符串 | err := errors.New("sql: record not found")          | 只是简单抛出错误信息使用标准包，errors.New方式 |
| ----- | --------------------------------------------------- | ---------------------------- |
| 动态字符串 | err := fmt.Errorf("sql: record not found, %s", sql) |                              |
err := errors.Errorf("sql: record not found, %s", sql) | 需要动态拼凑错误信息使用fmt.Errorf或者errors.Errorf方式，格式化字符串 |
1. 错误封装

遇到下游函数抛出的错误，有两种情况处理：

- 原样返回错误
- 需要封装错误，添加上下文信息，使用**github.com/pkg/errors包的**errors.WithMessage

如果没有要添加的其他上下文，则按原样返回原始错误。 如果需要添加上下文，**为了不破坏原始错误的信息，应使用errors.WithMessage方式，替代格式化字符串的方式**

| Good | Bad |
| --- | --- |
| s, err := store.New()
err1 := errors.WithMessage(err, "new store") | s, err := store.New()
err1 := fmt.Errorf("new store: %v", err) |
| 在传递错误过程中，可以保留原始错误；而且错误信息会向上渗透，形成堆栈 | 传递错误过程中，丢失了原始错误的其他信息和类型，对于上层的错误处理不友好 |
1. 复杂的错误封装

以上方式的错误封装，只能影响到错误信息的返回，有些场景则需要利用错误承载更多的信息，在上层进行错误特定字段的判断，比如访问第三方系统，需要承载返回的message和code，以便于后续在上层判断第三方系统的返回。

这种场景就需要实现标准包error接口，自定义错误，以下为例子

```
type ThirdPartyError struct {
    ErrCode int64
    ErrMsg  string
}

func (e *ThirdPartyError) Error() string {
    return fmt.Sprintf("errCode:%d, errMsg:%s", e.ErrCode, e.ErrMsg)
}

func NewThirdPartyError(code int64, message string) *ThirdPartyError {
    return &ThirdPartyError{
        ErrCode: code,
        ErrMsg: message
    }
}

err := NewThirdPartyError(code, msg)

```

## 错误处理

1. 一次处理错误

**无论调用方如何处理错误，它通常都应该只处理每个错误一次**。如果错误是可恢复的，则记录错误并正常降级。调用方不应该记录错误然后返回

例如，考虑以下情况：

| Description | Code |
| --- | --- |
| Bad: 记录错误并将其返回堆栈中的调用程序可能会对该错误采取类似的操作。这样做会在应用程序日志中造成大量噪音，但收效甚微。 | u, err := getUser(id)
if err != nil {
  // BAD: See description
  log.Printf("Could not get user %q: %v", id, err)
  return err
} |
| Good: 将错误换行并返回堆栈中更靠上的调用程序将处理该错误。 | u, err := getUser(id)
if err != nil {
  return errors.WithMessage(err, fmt.Sprintf("get user %q: %w", id))
} |
| Good: 记录错误并正常降级如果操作不是绝对必要的，我们可以通过从中恢复来提供降级但不间断的体验。 | if err := emitMetrics(); err != nil {
  // Failure to write metrics should not
  // break the application.
  log.Printf("Could not emit metrics: %v", err)
} |
1. 错误匹配处理

调用方了解错误的特点，需要进行错误的匹配，根据不同的方式去处理

例如，考虑以下情况：

| Description | Code |
| --- | --- |
| Good: 匹配普通error使用errors.Is或者errors.As方式匹配error | result, err := s.getGoods()
if err != nil {
   if errors.Is(err, ErrGoodsNotFound) {
       // handle specifically
   } else {
       return err
   } 
} |
| Good: 匹配自定义error使用断言方式匹配自定义error | result, err := s.getGoods()
if err != nil {
   switch oe := errors.Cause(err).(type) {
   case *MyError:
    // handle specifically
   default:
    // unknown error
   }
} |
| Good: controller层匹配error，透传error信息controller层识别特定的error，再透传返回前端的信息。返回前端的code和message，需要来源于error时，可以使用这种方式 | result, err := s.getGoods()
if err != nil {
   switch oe := errors.Cause(err).(type) {
   case *MyError:
       c.Error(e.Code(), err, e.Message()) // 透传code和message
       c.Error(e.Code(), err) // 只透传code
       c.Error(errs.ErrSystem, err， e.Message()) // 只透传message
   } else {
   default:
       c.Error(errs.ErrSystem, err)
   }
} |
1. 第三方系统处理错误

由于调用第三方系统，除了返回错误之外，还包含错误码，所以应该使用更加优雅的方式去处理

例如，考虑以下情况：

| Description | Code |
| --- | --- |
| Bad: 返回参数包含code调用方的错误处理不一定关注code，但是需要往上层继续抛出code，使得整个调用链都要透传code | func Get(req interface{}, path string) (interface{}, int64, error) {
    resp, err := client.Get(req, path)
    if err != nil {
        return nil, errors.SystemErrCode, err 
    }
    if resp.Code != 200 {
        return nil, resp.Code, errors.New(resp.Message)
    }
    // other handle
} |
| Bad: 返回参数只有error，但是不包含code信息这种方式对调用方友好，因为调用方只需要error判断，然后往上抛即可。但是上层可能需要关注code，这种处理使得错误信息缺失 | func Get(req interface{}, path string) (interface{}, error) {
    resp, err := client.Get(req, path)
    if err != nil {
        return nil, err 
    }
    if resp.Code != 200 {
        return nil, errors.New(resp.Message)
    }
    // other handle
} |
| Good: 返回参数只有error，error封装code这种方式对调用方友好，因为调用方只需要error判断，然后往上抛即可。同时对于上层，可以通过特定的识别，重新获取code | func Get(req interface{}, path string) (interface{}, error) {
    resp, err := client.Get(req, path)
    if err != nil {
        return nil, error
    }
    if resp.Code != 200 {
        // 封装的code，需要加上系统码
        return nil, cerror.NewThirdPartyError(
        fmt.Sprintf("%s%d", sysCode, resp.Code), resp.Message)
    }
    // other handle
} |

## 错误命名

对于存储为全局变量的错误值， 根据是否导出，使用前缀 `Err` 或 `err`

```
var (
  // 导出以下两个错误，以便此包的用户可以将它们与 errors.Is 进行匹配。
  ErrBrokenLink = errors.New("link is broken")
  ErrCouldNotOpen = errors.New("could not open")

  // 这个错误没有被导出，因为我们不想让它成为我们公共 API 的一部分。 我们可能仍然在带有错误的包内使用它。
  errNotFound = errors.New("not found")
)

```

对于自定义错误类型，请改用后缀 `Error`。

```
// 同样，这个错误被导出，以便这个包的用户可以将它与 errors.As 匹配。

type NotFoundError struct {
  File string
}

func (e *NotFoundError) Error() string {
  return fmt.Sprintf("file %q not found", e.File)
}

// 并且这个错误没有被导出，因为我们不想让它成为公共 API 的一部分。 我们仍然可以在带有 errors.As 的包中使用它。
type resolveError struct {
  Path string
}

func (e *resolveError) Error() string {
  return fmt.Sprintf("resolve %q", e.Path)
}

```

## 错误包管理

- ``component/cerror``为规范后的错误包
- 自定义错误和导出的错误统一放在该包
- 错误处理的通用函数（匹配，wrap, unwrap）统一放在该包
